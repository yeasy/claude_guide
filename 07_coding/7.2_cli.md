## 7.2 Claude Code CLI 入门与基础运维

**Claude Code** 是 Anthropic 推出的专为开发者设计的终端工具。它不仅仅是一个 API 包装器，而是一个具有 **Agentic Capabilities** 的编程助手——可以在授权范围内读取和编辑文件、运行终端命令，并辅助 Git 工作流。

### 7.2.1 安装与认证

#### 安装

通过 npm 安装：
```bash
npm install -g @anthropic-ai/claude-code
```

#### 认证

首次运行需要登录：
```bash
claude auth login
```
CLI 会打开浏览器引导完成 OAuth 授权；也可以在首次运行 `claude` 时按提示登录。可用 `claude auth status` 检查当前登录状态；脚本和 CI 场景不要复用交互式登录流程，应按官方 CLI 文档使用 `claude setup-token` 和专门的环境变量机制。

### 7.2.2 运行方式、认证与 settings

#### 常见运行方式

Claude Code 的 CLI 入口以官方命令和参数为准，不需要依赖内部运行路径或实现推断来判断行为：

| 方式 | 用途 | 示例 |
| :--- | :--- | :--- |
| 交互会话 | 启动终端内对话 | `claude` |
| 带初始提示的交互会话 | 进入会话前先发送一个任务 | `claude "explain this project"` |
| Print / SDK 模式 | 非交互式执行后退出，适合脚本和 CI | `claude -p "explain this function"` |
| 管道输入 | 把标准输入交给 Claude 处理 | `cat logs.txt \| claude -p "explain"` |
| 继续或恢复会话 | 继续最近会话，或按会话 ID/名称恢复 | `claude -c`、`claude -r "<session>"` |

#### 登录与认证

首次使用通常运行 `claude` 或显式执行登录命令：

```bash
claude auth login
claude auth status --text
```

`claude auth login` 可配合 `--email` 预填邮箱、`--sso` 强制 SSO、`--console` 使用 Anthropic Console 账号进行 API 计费登录。`claude auth status` 会显示认证状态；默认输出 JSON，`--text` 输出便于人工阅读。

脚本、CI 或无法打开浏览器的环境，不应复用本机交互式登录状态。官方提供 `claude setup-token` 生成长期 OAuth token，命令只把 token 打印到终端，不会保存；将其配置为 `CLAUDE_CODE_OAUTH_TOKEN` 后再运行脚本：

```bash
claude setup-token
export CLAUDE_CODE_OAUTH_TOKEN=your-token
```

如果同时存在多种凭据，Claude Code 按官方认证文档解析：云服务提供商凭据（如 Bedrock、Vertex AI、Foundry）优先，其次是 `ANTHROPIC_AUTH_TOKEN`、`ANTHROPIC_API_KEY`、`apiKeyHelper`、`CLAUDE_CODE_OAUTH_TOKEN`，最后才是订阅账号的 OAuth 登录状态。排查认证问题时，先确认当前是否设置了会覆盖订阅登录的环境变量。

#### settings 层级

`settings.json` 是 Claude Code 的正式配置机制，适合放权限、环境变量、hooks、插件等可执行配置。常见位置如下：

| 作用域 | 文件位置 | 用途 |
| :--- | :--- | :--- |
| User | `~/.claude/settings.json` | 当前用户的全局偏好 |
| Project | `.claude/settings.json` | 随项目提交、团队共享 |
| Local | `.claude/settings.local.json` | 当前项目的个人本地覆盖，通常不提交 |

settings 的合并优先级为：

1. Managed settings
2. 命令行参数
3. Local：`.claude/settings.local.json`
4. Project：`.claude/settings.json`
5. User：`~/.claude/settings.json`

常用命令行参数：

```bash
# 只加载指定 settings 来源
claude --setting-sources user,project

# 从文件或 JSON 字符串额外加载 settings
claude --settings ./settings.json
```

`--setting-sources` 接受逗号分隔的 `user`、`project`、`local`，用于控制本次启动读取哪些普通 settings 来源。Managed settings 和命令行参数仍按其自身层级生效。

### 7.2.3 核心命令

#### 交互模式

直接运行 `claude` 进入对话模式：

```bash
$ claude
> 帮我分析一下当前目录的结构。
(Claude runs `ls -R`)
当前目录包含 src, tests 和 docs...

> 把 src/utils.py 里的日志库从 logging 换成 structlog。
(Claude reads src/utils.py, edits, shows diff)
```

#### 单次任务

```bash
claude "generate a commit message for the current changes"
claude "explain main.go"
claude "fix strict null checks"
```

#### Unix 哲学与管道：Piping

Claude Code 拥有高度的可组合性，遵循 Unix 哲学。你可以将日志、命令输出通过管道符 (`|`) 传给 Claude，进行串联处理：

```bash
# 监控日志并让 Claude 进行异常告警
tail -f app.log | claude -p "Slack me if you see any anomalies"

# 结合持续集成进行自动化处理
claude -p "translate new strings into French and raise a PR for review"

# 批量进行文件安全检查
git diff main --name-only | claude -p "review these changed files for security issues"
```

### 7.2.4 CLAUDE.md：项目记忆与上下文

`CLAUDE.md` 用来给 Claude Code 提供项目说明、约定和常用命令。它不是 `settings.json`，不适合承载权限、环境变量、API provider 等可执行配置；这些应写入 settings。

#### 什么是 CLAUDE.md？

`CLAUDE.md` 是 Claude Code 的记忆文件。它的内容会作为用户消息上下文提供给模型，而不是写入 API 的 system prompt 字段，也不是不可违背的系统级指令。需要强制执行的安全策略、工具权限和沙箱约束，应使用 settings、Managed settings 或外部系统约束来实现。

#### 文件位置

| 位置 | 作用域 | 说明 |
| :--- | :--- | :--- |
| `~/.claude/CLAUDE.md` | User | 当前用户所有项目通用 |
| 项目 `CLAUDE.md` | Project | 放在项目根或上级目录，适合团队共享 |
| 项目 `.claude/CLAUDE.md` | Project | 与其他 `.claude/` 项目配置放在一起 |
| `CLAUDE.local.md` | Local | 当前项目或目录层级的个人说明，通常加入 `.gitignore` |

#### 加载机制

Claude Code 会从当前工作目录向上遍历目录树，查找每一层的 `CLAUDE.md` 和 `CLAUDE.local.md`。启动时会加载当前目录层级之上的文件；当前工作目录下子目录里的说明文件，会在 Claude 读取那些子目录中的文件时按需加入上下文。

所有发现的文件会拼接进上下文，而不是按“高优先级覆盖低优先级”的方式替换。位于同一目录时，`CLAUDE.local.md` 会追加在 `CLAUDE.md` 后面，因此本地个人说明在该层级最后被读到。

这几个结论对写作最重要：

- `CLAUDE.md` 适合写项目结构、编码规范、测试命令、发布步骤等背景说明。
- `CLAUDE.local.md` 适合写个人本机路径、沙箱 URL、本地测试数据等不应提交的信息。
- 冲突越少越好；不要把它当成一套可覆盖、可继承的强制策略系统。
- 如果要确认当前加载了哪些记忆文件，可在 Claude Code 中使用 `/memory`。

#### 引用其他文件

`CLAUDE.md` 可以用 `@` 引用其他文件，把长说明拆成更小的主题文档：

```markdown
@docs/coding-standards.md
@.claude/shared-rules.md
```

引用应保持清晰、可审查。大型项目更推荐把长期通用规则放入 `.claude/rules/`，并用路径匹配让特定规则只在处理对应文件时进入上下文。

#### CLAUDE.local.md：本地说明

`CLAUDE.local.md` 适合保存不会影响团队其他成员的个人偏好：

- 本机专用的构建命令或服务地址
- 个人常用测试数据
- 临时实验说明

不要在 `CLAUDE.local.md` 中保存密钥本身。认证材料应走系统凭据、环境变量、settings 或专门的密钥管理工具。

#### 示例 CLAUDE.md

````markdown
# Project Context

When working with this codebase, prioritize readability over cleverness.

## About This Project

FastAPI REST API for user authentication. Uses SQLAlchemy and Pydantic.

## Key Directories

- `app/models/` - database models
- `app/api/` - route handlers
- `app/core/` - configuration and utilities

## Standards

- Type hints required on all functions
- pytest for testing (fixtures in `tests/conftest.py`)
- PEP 8 with 100 character lines

## Common Commands

```bash
uvicorn app.main:app --reload  # dev server
pytest tests/ -v               # run tests
```

## Notes

All routes use `/api/v1` prefix. JWT tokens expire after 24 hours.
````

#### 使用 /init 自动生成

运行 `/init` 命令可以自动分析项目并生成初始 `CLAUDE.md`：

```bash
cd your-project
claude
/init
```

Claude 会检查 package.json、README、配置文件等，生成一份针对你项目的配置文件。

#### 自动记忆

Claude Code 还提供自动记忆能力，用来跨会话保存可能复用的信息，例如常用构建命令、调试发现、架构备注和工作流偏好。它不是每次会话都写入内容，也不替代团队维护的 `CLAUDE.md`；对团队必须共享的规则，仍应显式写入项目记忆或 settings。

#### Hooks：拦截命令体系

为了在流程中更自如地植入代码质量检测，Claude Code 引入了 **Hooks** 机制：
你可以约定让 Shell 命令在 Claude 发生相关行为前/后自动执行。例如修改完任何文件后必须走一遍代码格式化（自动运行 Prettier/Black）；再比如可以在让 Claude 代码修改准备 Commit 提交之前执行全量的 Lint 检测，确保不引入新的问题。

### 7.2.5 自定义 Slash Commands

将常用的提示词封装为自定义命令，避免重复输入。

> **注意**：自定义命令功能可用性取决于当前 Claude Code 版本。建议参考官方文档了解最新支持。

#### 创建自定义命令

在项目根目录创建 `.claude/commands/` 目录并添加 Markdown 文件：

```bash
mkdir -p .claude/commands
```

创建 `.claude/commands/security-review.md`：

```markdown
# Security Review

Analyze the provided code for security vulnerabilities:

## Areas to Check
- SQL injection risks
- XSS vulnerabilities
- Authentication/authorization issues
- Sensitive data exposure
- Input validation

## Output Format
For each issue:
1. **Severity**: Critical/High/Medium/Low
2. **Location**: File and line number
3. **Issue**: Description
4. **Fix**: Recommended solution

**Code to review:**
$ARGUMENTS
```

#### 使用自定义命令

```bash
> /security-review src/auth/login.py
```

`$ARGUMENTS` 会被替换为命令后的参数。

> **使用提示**：内置命令（如 `/clear`、`/compact`）和自定义 commands/skills 的处理方式不同。排查斜杠命令问题时，应优先参考当前版本的官方 Commands 文档。

### 7.2.6 上下文管理技巧

#### 使用 /clear 重置上下文

长时间对话会累积无关上下文。在切换任务时使用：

```bash
> /clear
```

这会清除对话历史，但保留 `CLAUDE.md` 配置。

#### 使用 Subagent 隔离上下文

对于需要不同视角的任务（如实现后的安全审查），指示 Claude 使用子代理：

```bash
> 使用一个子代理对刚才实现的代码进行安全审查
```

子代理拥有独立的上下文，不会被前序开发思路影响。

> **注意**：Remote Control 的能力边界和可用命令可能随版本变化。使用 `claude remote-control` 或 `--remote-control` 前，应以当前官方文档和本机 `claude --help` 输出为准。

### 7.2.7 权限模式

Claude Code 用权限规则和权限模式共同控制工具调用。权限规则可以写在 settings 的 `permissions.allow`、`permissions.ask`、`permissions.deny` 中；模式决定未被规则明确处理的工具调用如何审批。

启动时可用 `--permission-mode` 指定模式：

```bash
claude --permission-mode plan
```

官方支持的取值如下：

| 模式 | 行为 |
| :--- | :--- |
| `default` | 标准行为：首次使用需要审批的工具时询问 |
| `acceptEdits` | 自动接受工作目录或额外目录内的文件编辑和常见文件系统命令 |
| `plan` | 计划模式：无需提示可读文件；可按默认审批规则运行探索命令，但不修改源码 |
| `auto` | 通过后台安全检查自动审批符合请求意图的工具调用 |
| `dontAsk` | 不弹出审批；除已预先允许的工具和只读命令外，其余自动拒绝 |
| `bypassPermissions` | 跳过大多数权限提示；仅应在容器、虚拟机等隔离环境使用 |

`--dangerously-skip-permissions` 等价于 `--permission-mode bypassPermissions`。如果只想把 `bypassPermissions` 加入 `Shift+Tab` 模式循环而不是直接进入，可使用 `--allow-dangerously-skip-permissions`。

不要把 `bypassPermissions` 理解为“安全自动驾驶”。它会跳过权限提示和安全检查，但对受保护路径的写入仍会提示，例如 `.git`、`.vscode`、`.idea`、`.husky`、大部分 `.claude` 目录以及 `.mcp.json`、`.claude.json` 等关键文件。生产仓库中更常见的做法是使用 `default`、`acceptEdits`、`plan` 或 `auto`，并通过 settings 明确 allow/deny 规则。

### 7.2.8 以官方接口为准

Claude Code 更新频繁，CLI 章节应依赖官方文档公开承诺的接口：命令、flags、settings、permissions、memory、hooks、MCP 和 SDK。第三方分析得到的类名、内部状态字段、非公开实现细节、提示词拼接细节、缓存边界等实现描述不适合作为使用指南，因为它们可能随版本变化，也不是产品契约。

排查问题时优先检查这些稳定入口：

- `claude --version`：确认本机版本。
- `claude auth status --text`：确认登录与凭据来源。
- `/config`：查看和修改交互式配置。
- `/permissions`：查看权限规则来自哪个 settings 文件。
- `/memory`：确认加载了哪些 `CLAUDE.md` / `CLAUDE.local.md`。
- `--settings`、`--setting-sources`、`--permission-mode`：复现实验环境时显式指定配置来源和权限模式。

如果确实需要调整系统提示词，应使用官方系统提示词参数：`--system-prompt`、`--system-prompt-file`、`--append-system-prompt`、`--append-system-prompt-file`。`CLAUDE.md` 是项目记忆上下文，不应被描述为 system prompt 的一部分。
