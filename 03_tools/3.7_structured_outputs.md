## 3.7 结构化输出：保证响应格式的可靠性

在生产环境中，一次格式错误就可能引发级联故障：JSON 解析失败、工具调用参数不匹配、多 Agent 通信中断。Claude Messages API 的结构化输出分成两类互补能力：用 `output_config.format` 约束 Claude 最终回复的 JSON 形状，用 `strict: true` 约束工具调用参数。

> 核验日期：2026-04-27。Anthropic Structured Outputs 文档说明 JSON outputs 使用 `output_config: {"format": {"type": "json_schema", "schema": ...}}`，响应文本位于 `response.content[0].text`；Strict Tool Use 使用 `tools[].input_schema` 与 `strict: true` 约束工具参数。

### 3.7.1 为什么需要结构化输出

仅靠提示词要求 Claude “以 JSON 格式返回”并不适合生产系统：模型可能输出 Markdown 包装、遗漏必填字段，或把数值、布尔值、数组等类型写错。应把结构约束放进 JSON Schema，让 API 按 schema 生成可解析的 JSON；如果 Claude 还需要调用工具，再用 strict tool use 保证工具参数合法。

### 3.7.2 最小可运行示例

下面的示例使用 Messages API 的 JSON outputs，直接要求最终回复匹配 JSON Schema：

```python
# pip install anthropic
import json
from anthropic import Anthropic

client = Anthropic()

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=512,
    messages=[
        {
            "role": "user",
            "content": (
                "Extract metadata from this note: The Left Hand of Darkness "
                "by Ursula K. Le Guin was published in 1969. Topics: "
                "science fiction, gender, anthropology."
            ),
        }
    ],
    output_config={
        "format": {
            "type": "json_schema",
            "schema": {
                "type": "object",
                "properties": {
                    "title": {"type": "string"},
                    "author": {"type": "string"},
                    "published_year": {"type": "integer"},
                    "topics": {
                        "type": "array",
                        "items": {"type": "string"},
                    },
                },
                "required": ["title", "author", "published_year", "topics"],
                "additionalProperties": False,
            },
        }
    },
)

result = json.loads(response.content[0].text)
print(result)
```

关键点：

- `output_config.format` 使用 JSON Schema，定义 Claude 最终回复必须匹配的数据结构。
- 响应正文是合法 JSON 字符串，位于 `response.content[0].text`，客户端再用 JSON parser 读取。
- Python SDK 的 `client.messages.parse()` 可以用 Pydantic model 做便捷封装；底层仍会转换为 `output_config.format`。
- 不要把其他 API 的同名或近似参数照搬到 Anthropic Messages API。

### 3.7.3 Strict Tool Use

如果目标不是“最终回复 JSON”，而是“生成工具参数”，应把 schema 放在工具定义中，并开启 `strict: true`：

```python
response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=512,
    tools=[
        {
            "name": "extract_book_metadata",
            "description": "Extract book metadata from the user's text.",
            "strict": True,
            "input_schema": {
                "type": "object",
                "properties": {
                    "title": {"type": "string"},
                    "author": {"type": "string"},
                },
                "required": ["title", "author"],
                "additionalProperties": False,
            },
        }
    ],
    tool_choice={"type": "tool", "name": "extract_book_metadata"},
    messages=[{"role": "user", "content": "The Left Hand of Darkness, Ursula K. Le Guin"}],
)

tool_use = next(block for block in response.content if block.type == "tool_use")
tool_args = tool_use.input
```

JSON outputs 控制 Claude “说什么格式”，strict tool use 控制 Claude “怎样调用工具”。两者可以在同一个请求中组合：先调用工具时保证参数合法，最终回答时再保证回复 JSON 合法。

### 3.7.4 适用场景

**数据提取**：从合同、发票、简历、读书笔记等非结构化文本中提取结构化字段，降低下游解析失败率。

**多 Agent 通信**：Agent 之间通过 schema 化消息传递状态、任务和结果，减少格式漂移。

**工具参数生成**：为搜索、工单、订单、数据库查询等工具生成严格匹配 schema 的参数，此时使用 strict tool use。

### 3.7.5 注意事项

- JSON outputs 会增加少量输入 token，并且 schema 首次编译会带来额外延迟；编译后的 grammar 会被缓存。
- 拒绝回答、达到 `max_tokens` 等情况可能导致输出不符合 schema，客户端仍要检查 `stop_reason` 和解析错误。
- `tool_choice` 的 `auto` 允许 Claude 自行决定是否调用工具；`none` 禁止工具调用；`any` 强制调用任一工具；`tool` 强制调用指定工具。
- `tool_choice: any` 和指定 `tool` 不支持与 extended thinking 同时使用；需要 extended thinking 时使用 `auto` 或 `none`。
- Claude Mythos Preview 不支持 forced tool use；不要在该模型上使用 `tool_choice: any` 或指定 `tool`。
- 对有副作用的真实业务工具也可以使用 `strict: true`，但执行工具前仍应做权限、幂等和业务校验。
- schema 过深或可选字段过多会增加复杂度；复杂输出应拆分成更小的 schema。
