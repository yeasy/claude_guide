## 8.4 扩展思考：深度推理模式

2024 年底，OpenAI o1 和 Claude 的新特性引发了关于 **System 2 Thinking** 的热潮。
所谓的 “Extended Thinking” (或称 Thinking Mode)，是指模型在输出最终答案之前，能够在后台进行不可见的、长时间的思维推理。

### 8.4.1 什么是 System 1 vs System 2？

*   **System 1 (快思考)**: 直觉式反应。
    *   Example: "2+2=?" -> "4"。
    *   Standard LLM Mode.
*   **System 2 (慢思考)**: 逻辑推理，逐步验证。
    *   Example: “计算 3491 x 231” -> (列竖式...) -> “806421”。
    *   Extended Thinking Mode.

在 Extended Thinking 模式下，Claude 可能会花费 10 秒甚至几分钟来生成数千个 Hidden Tokens，用于自我辩论、尝试错误路径并回溯，最终只输出一个高质量的答案。

### 8.4.2 启用与控制

虽然具体 API 形式随时间演进，但逻辑通常如下：

```python
response = client.messages.create(
    model="claude-4-5-sonnet-20250929", # 支持 thinking 的最新版本
    max_tokens=8192,
    thinking={ 
        "type": "enabled", 
        "budget_tokens": 4096  # 分配给思考过程的 Token 预算
    }, 
    messages=[{"role": "user", "content": "设计一个高并发秒杀系统的架构..."}]
)
```

用户通常 **看不到** 这 4096 个思考 Token（或者只能看到折叠的摘要），但最终结果的质量会显著提升。

### 8.4.3 工作流程解密

当启用了 Thinking Mode，Claude 的内部独白可能长这样：

> *(Hidden Thinking Block)*
> User wants a high-concurrency system.
> *Initial thought*: Use Redis + MySQL.
> *Critique*: Simple Redis might act as a bottleneck if not sharded.
> *Alternative*: Explore Lua scripts for atomicity.
> *Check*: What about message queues? Kafka vs RabbitMQ?
> *Decision*: Let's propose a 3-layer architecture: CDN -> Nginx L7 -> MQ -> Consumer -> DB.
> *Refinement*: Need to mention cache consistency strategies (Cache-Aside vs Write-Through).
> ...

最终输出：
“这是一个推荐的三层架构方案...”（逻辑严密，无漏洞）

### 8.4.4 适用场景

并非所有任务都需要慢思考。

| 场景 | Extended Thinking | 理由 |
| :--- | :--- | :--- |
| **创意写作** | ❌ NO | 需要发散思维，过度逻辑化会扼杀灵感。 |
| **日常闲聊** | ❌ NO | 用户不能忍受 10 秒延迟。 |
| **数学证明** | ✅ YES | 需要严密的逻辑推导。 |
| **复杂代码架构** | ✅ YES | 需要权衡多种设计模式的利弊。 |
| **法律/医疗建议** | ✅ YES | 容错率极低，需要自我审查。 |

### 8.4.5 推理的前沿方向：超越语言链

Extended Thinking 本质上仍是基于自然语言的串行推理。研究前沿正在探索两个可能打破语言低带宽瓶颈的方向：**隐空间推理**（在高维向量空间而非语言序列中进行推理，效率有指数级提升潜力）和**并行推理**（拆解独立子问题并发执行，突破串行延迟）。这些方向暗示：**下一代推理能力的提升可能不再来自"想得更久"，而是来自"想得更高效"**。

> 关于隐空间推理（Coconut）、并行推理以及推理时扩展上界的技术细节，参见姊妹篇《大语言模型原理》14.6.6 节。

### 8.4.6 成本分析

Extended Thinking 虽然提升了质量，但需要理解其成本结构。

#### 思考 Token 的计费方式

关键点：**思考 Token 按照 OUTPUT 价格计费**，而非 INPUT 价格。

**成本公式**

```
Total Cost = Input Tokens × Input Rate + Thinking Tokens × Output Rate + Output Tokens × Output Rate
```

**具体例子**（基于 Claude Opus 4.6 定价）

假设：
- Input: 2000 tokens
- Thinking: 5000 tokens
- Output: 500 tokens

```
Input cost:     2000 × $5/M = $0.010
Thinking cost:  5000 × $25/M = $0.125  ← 注意：用 Output 价格！
Output cost:    500  × $25/M = $0.0125
────────────────────────────────
Total:                         $0.1475
```

相比无思考的同样输出（仅 Output cost $0.0125），额外成本为 $0.135，即增加了 **1080%**！

#### 成本优化建议

**1. 选择合适的 budget_tokens**

```python
response = client.messages.create(
    model="claude-opus-4-6-20251101",
    max_tokens=2000,
    thinking={
        "type": "enabled",
        "budget_tokens": 2000  # 不要盲目设置 8000+
    },
    messages=[...]
)
```

根据任务复杂度调整：
- **简单任务**: 500-1000 tokens
- **中等任务**: 1500-3000 tokens
- **复杂推理**: 4000-8000 tokens（仅在必要时）

**2. 成本-质量权衡**

对比分析：

| 场景 | 推荐方案 | 成本 | 质量提升 |
|------|--------|------|--------|
| **简单分类** | 不用 Extended Thinking | 基准 | 基准 |
| **代码审查** | budget=2000 | 基准+40% | +30% 精度 |
| **架构设计** | budget=4000 | 基准+80% | +60% 精度 |
| **复杂数学** | budget=6000 | 基准+150% | +85% 准确率 |

**3. 使用 budget_tokens 限制思考深度**

```python
# 案例：法律合同审查
response = client.messages.create(
    model="claude-opus-4-6-20251101",
    max_tokens=3000,
    thinking={
        "type": "enabled",
        "budget_tokens": 3000  # 限制最多 3000 thinking tokens
    },
    system="你是资深律师，审查以下合同...",
    messages=[
        {"role": "user", "content": contract_text}
    ]
)
```

在这种情况下，即使模型想思考更多，也会在 3000 tokens 处停止，从而控制成本。

#### ROI 分析：什么时候值得开启 Extended Thinking？

```
每个请求的额外成本 = Thinking Tokens × (Output Rate - Input Rate)
                  = 5000 × ($25 - $15) / 1M
                  = $0.05

若质量提升能减少重试次数：
  - 原本需要 3 次尝试（无思考）= $0.15
  - 用 Extended Thinking 一次成功 = $0.05
  → 节省 $0.10，值得！
```

**决策框架**

```python
def should_use_extended_thinking(task_type: str, expected_cost_per_attempt: float) -> bool:
    """
    判断是否应该启用 Extended Thinking
    """
    thinking_overhead = 5000 * 10 / 1_000_000  # 5000 thinking tokens, $10/M delta

    if task_type in ["math_proof", "complex_reasoning", "code_architecture"]:
        # 这些任务通常能从深思熟虑中获益
        # 质量提升能抵消额外成本
        return True

    if task_type in ["classification", "simple_qa", "summarization"]:
        # 简单任务，extended thinking 浪费钱
        return False

    # 按成功率计算 ROI
    success_rate_without_thinking = 0.7
    success_rate_with_thinking = 0.95
    avg_retries_without = 1 / success_rate_without_thinking
    avg_retries_with = 1 / success_rate_with_thinking

    total_cost_without = expected_cost_per_attempt * avg_retries_without
    total_cost_with = (expected_cost_per_attempt + thinking_overhead) * avg_retries_with

    return total_cost_with < total_cost_without
```

### 8.4.7 对 Agent 架构的影响

Extended Thinking 改变了设计 Agent 的方式。

#### 减少 Loop 次数

以前可能需要用 ReAct 模式让 Agent 显式地思考 5 轮。
现在，可以把这 5 轮思考内化为一次 Extended Thinking Call。
**External Loop -> Internal Thought**。
不仅速度更快（减少网络 RTT），而且模型在内部状态下的注意力机制更集中。

#### 混合策略

在 Multi-Agent 系统中，可以让 **Planner Agent** 开启 Extended Thinking（负责深思熟虑定计划），而 **Executor Agent** 使用普通模式（负责快速执行）。
